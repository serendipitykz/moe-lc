KV Cache 是长上下文推理最核心的内存瓶颈。MLA 从架构层面解决它；SnapKV 和 H2O 是推理时的动态驱逐；StreamingLLM 是极简无限流推方案。三种思路层次不同，可以叠加使用。 --- 我们现在来看 KV Cache 管理这个主题——它是长上下文推理的核心瓶颈，也是几乎所有生产部署都必须面对的工程问题。 先感受一下规模：LLaMA-3 70B 在 128K 上下文下，KV Cache 需要约 40 GB，已超过单张 A100。Mixtral 8×7B 约 16 GB。而 DeepSeek-V2 用了 MLA，只需约 6 GB——差距是数量级的。 **MLA（Multi-head Latent Attention）**，DeepSeek-V2。 MLA 的解法是架构级低秩压缩。标准 MHA 要缓存每层每头的完整 K 和 V（DeepSeek-V2 的 d_model=5120）。MLA 改为：给每个 token 的隐状态 h_t（dim=5120），先用下投影矩阵 W^DKV 压缩到一个低维潜在向量 c_KV（dim=512）。推理时只缓存这个 512 维的向量，需要 K、V 时再用上投影矩阵 W^UKV 还原为完整维度。 这把 KV Cache 从 5120×2 = 10240 维压缩到 512 维，约 12 倍压缩。DeepSeek-V2 60 层下，每 token 从 ~480 KB 降到 ~48 KB，128K 上下文从 ~40 GB 降到 ~6 GB。 一个限制：RoPE 位置编码不能被纳入低秩压缩（RoPE 需要精确的位置感知，而低秩 c_KV 丢失了位置信息）。MLA 用"解耦 RoPE"处理这个问题——为 RoPE 单独保留一组小 head，这些 head 的完整 K 向量仍需缓存。 **SnapKV** [Li et al., 2024]：三步流程。 在 prompt 末尾设一个"观察窗口"，在这个窗口内计算 query 对所有 prefix KV 的注意力分数——这些分数是 prefix 中每个 token 被"关注过多少次"的投票。用池化聚类算法保留高票的代表性 KV + 最近窗口。SnapKV 是 per-layer 粒度，不同层独立决策。 **H2O（Heavy Hitter Oracle）** [Zhang et al., 2023]：注意力分数的分布呈幂律——少数"重击者"token 累积了大部分注意力权重。动态维护 top-k 累积分数 + 最近性，只保留 20% 的 KV 预算即可近似全精度。 **Attention Sink + StreamingLLM** [Xiao et al., 2023]：一个意外的发现——前几个 token（位置 0–3）无论内容如何都会吸引异常高的注意力权重，因为 softmax 无法输出 0，模型把"多余的"权重倾倒在初始位置作为泄压阀。StreamingLLM 利用这个现象：只保留 Sink 窗口（前 K_s≈4 token）+ 最近窗口（后 K_r token），实现 O(1) 内存的无限流式推理，无需重训练。代价是中段信息完全丢失。
