长上下文推理的速度问题：传统投机采样在长上下文下失效，RAPID 和 SuffixDecoding 是两种不同的解决路径。重点是投机采样失效的原因——这是一个值得深思的系统性问题。 --- 讲完了注意力机制本身的效率，我们来看推理速度的另一个角度：**投机采样（Speculative Decoding）**在长上下文下的问题和新的解决方案。 **为什么传统投机采样在长上下文下失效？** 投机采样的基本思路大家应该熟悉：用一个小的 draft 模型快速生成若干候选 token，然后用大的 target 模型批量验证，接受或拒绝。在短上下文场景下这很有效，典型加速 3-4x。 但在长上下文场景（128K 甚至 1M token），传统投机采样面临两个根本性问题： 第一，**KV Cache 内存压力**。长上下文下，target 模型的 KV Cache 本身就需要几十 GB。再加上 draft 模型的 KV Cache，两套缓存同时占用内存，导致 batch size 被迫减小，整体吞吐量反而下降。 第二，**接受率崩溃**。draft 模型通常参数量比 target 模型小 5-10 倍，在短上下文下还能给出相近的分布。但在长上下文下，draft 模型对这么长的 prefix 的理解已经严重偏离 target 模型——它的"注意力能力"不足以处理长序列，生成的 token 会偏离分布，导致大量被拒绝，加速效果消失。 **RAPID：用 RAG 替代 draft 模型** RAPID 的洞察是：为什么 draft 模型对长序列理解差？因为它需要处理全部 prefix。但如果我们用 RAG 从长上下文里检索出最相关的 4K token，再让 draft 用这个**短**的检索结果来预测，draft 的准确率就能大幅提升。 理论依据：4K 精确检索 ≈ 128K 全量上下文的准确率。实验表明，对于大多数 QA 任务，检索到正确片段后，4K 上下文模型的答案和 128K 全量模型高度一致。 RAPID 修改了传统的接受准则：加入检索置信度作为权重，接受条件变为 r ≤ min(1, p_φ(x_i|C) / q_ψ(x_i|C_S))，其中 C 是完整上下文，C_S 是检索子集。 **SuffixDecoding：从历史生成中找规律** SuffixDecoding 的思路完全不同。它的观察是：语言有大量的重复模式——特定的短语后面往往跟着相似的后续词。如果我们维护两棵后缀树：一棵来自当前推理会话的历史输出，一棵来自大型语言模型语料库，就可以在推理时查找"当前上下文末尾的后缀"对应的历史续写，生成一棵投机树。然后让 target 模型批量验证这棵树，取最优分支。 SuffixDecoding 完全不需要 draft 模型，也不需要额外的 KV Cache——它是纯粹基于历史记录的模式匹配。实测约 3 倍加速。 总结来说，长上下文推理加速是一个系统级问题，不只是算法问题——需要内存管理、并行策略和推理策略的联合优化。

传统投机采样的失效

① KV Cache 内存压力

128K 上下文下，target + draft 两套 KV Cache 同时占显存，强制减小 batch size，整体吞吐量下降。

② 接受率崩溃

Draft 模型对超长 prefix 的理解严重偏离 target 模型——生成 token 偏离分布，大量被拒绝，加速效果消失。

RAPID：RAG 充当草稿模型

用 RAG 检索器替代 draft 模型——4K 精确检索片段 ≈ 128K 全量上下文的预测准确率。

接受准则纳入检索置信度：\(r \le \min(1,\, p_\phi(x \mid C) / q_\psi(x \mid C_S))\)。

关键洞察

4K 精确检索 ≈ 128K 全量上下文准确率（大多数 QA 任务）；draft 始终在短上下文下运行，接受率稳定。

SuffixDecoding：后缀树投机

无 draft 模型——从推理历史和语料库构建后缀树，生成投机树后批量验证。

优势

无额外 KV Cache（无 draft 模型），无需知道上下文长度

加速

~3× over incremental decoding
