All-to-all 通信和 KV Cache 是部署 MoE 的两大"工程噩梦"，引发听众对实际系统复杂度的思考。

MoE 推理的工程挑战

部署 MoE 模型的三大工程挑战

01

专家并行与通信开销

  • Expert Parallelism： 将 \(N_{\text{exp}}\) 个专家分布到多张 GPU，每个专家驻留在一个设备上

  • 每个 MoE 层需要 all-to-all 通信： token → 找到对应专家的设备 → 计算 → 返回结果

  • 通信延迟随专家数线性增长，可占总推理时间的 20–40%

  • 缓解： EP + Tensor Parallelism 混合并行；减少 MoE 层数

02

KV Cache 的显存压力

  • \(L=128\text{K}\) 时，Llama-3-70B 的 KV Cache ≈ 344 GB ——远超模型参数本身

  • MoE 模型：KV Cache 同样随 \(L\) 线性增长，与 专家参数竞争显存

  • 解决方案： PagedAttention（虚拟内存分页）[Kwon et al., 2023]、KV Cache 量化、Sliding Window Attention

  • Trade-off： 截断历史（损失信息）vs. 保留（超显存预算）

03

专家 Cache 预热

  • 不同 batch 可能路由到不同专家 → 频繁换入换出专家参数（CPU ↔ GPU 交换）

  • 对于总参数远超显存的模型（如 236B DeepSeek-V2）， 专家 offloading 是必要的

  • 缓解： 基于路由统计 预测下一步激活专家， 提前 prefetch 到 GPU，隐藏换入延迟

  • 专家访问具有局部性规律（相似输入 → 相似路由）， 可利用此规律设计高命中率的专家 Cache 策略
