混合专家模型
与长上下文

Mixture of Experts & Long Context

CS2916-01 大语言模型  ·  ACM Class 2026  ·  StarryPurple

大家好，我是 StarryPurple，今天的报告题目是《混合专家模型与长上下文》。

在过去几年，大语言模型的发展呈现出两条高度相关的主线：一是模型参数规模的急剧扩张，从 GPT-2 的 15 亿参数一路飙升到如今数千亿量级；二是上下文窗口的持续拉长，从最初的 2K token 扩展到今天的百万量级。

MoE 在参数规模上给出了一个绕过算力瓶颈的优雅答案；长上下文技术则是在使用层面对模型能力的另一维拓展。今天的报告约四十分钟，分六个部分，以三个开放性假说收尾。

03

规模定律：越大越好？

Scaling Laws
  • Kaplan (2020)：$L \approx \bigl(N_c/N\bigr)^{0.076}$，性能随 $N$、$D$、$C$ 呈幂律增长，暂无明显上限
  • Chinchilla (2022)：计算最优时 $N_{\text{opt}} \propto C^{0.5}$，$D_{\text{opt}} \propto C^{0.5}$——token 数应约等于 20×参数量
  • GPT-3 仅用 300B token 训练 175B 模型 → 数据严重不足；Llama 3 修正为 15T token + 405B 参数
核心矛盾：参数量 $10\times$ ⇒ 推理成本 $10\times$。模型训练一次，但推理每次请求都要重做。
能否让模型「拥有」更多参数，却只「使用」其中一部分？
Chinchilla 最优分配
算力预算最优参数量最优 Tokens
$10^{21}$ FLOP~1B~20B
$10^{23}$ FLOP~10B~200B
$10^{24}$ FLOP~70B~1.4T ← Llama 2
$10^{25}$ FLOP~220B~4.4T ← Llama 3
推理 FLOPs/token ≈ $2N$：
175B Dense → 350 GFLOP/tok
Llama 3 405B → 810 GFLOP/tok

规模定律的核心结论是：更大的模型、更多的数据、更多的算力，都会带来可预测的性能提升，且暂无拐点。Kaplan 给出的参数量幂律指数约为 0.076，意思是参数量每增加 10 倍，loss 大约降低 0.076 个 nats。

Chinchilla 的贡献是：在固定算力下，参数量和训练 token 数应该同步扩大，比例约为 1:20。这意味着 GPT-3 的 300B token 对 175B 模型来说严重不足。Llama 3 用 15T token 训练 405B 模型，比例约为 1:37，比 Chinchilla 建议的还要更数据密集。

但无论怎么优化训练，推理成本的线性增长是无法回避的。右侧的表格显示，一个 405B 模型每生成一个 token 需要约 810 GFLOP 的计算——而服务大规模用户时，这个成本会直接转化为延迟和电费。这就是 MoE 要解决的问题。

04

稠密 vs 稀疏激活

Dense vs Sparse Activation

Dense（稠密）

  • 每个 token 激活全部参数
  • 推理 FLOPs ∝ 总参数量 $N$
  • KV cache 随序列长度线性增长，与 $N$ 无直接关联
  • 硬件利用率高，内存访问模式规整
  • 代表：GPT-4（~1T est.）、Llama 3（405B）
$$\text{FLOPs} \approx 2 \cdot N \cdot T$$

Sparse MoE

  • 每个 token 仅激活 $k$ 个专家（通常 $k \ll E$）
  • 推理 FLOPs ∝ 激活参数量 $N_{\text{act}}$，$N_{\text{act}} \ll N$
  • 代价：所有专家须全驻显存；分布式部署需额外通信
  • 训练时梯度稀疏，需辅助损失维持稳定
  • 代表：Mixtral 8×7B（47B→13B act）、DeepSeek-V2
$$\text{FLOPs} \approx 2 \cdot N_{\text{act}} \cdot T \quad (N_{\text{act}} \ll N)$$
以 Dense 级别的推理 FLOPs，撬动远大于 Dense 的参数容量——参数量 ≠ 激活参数量

这张对比图要强调的核心区分是：「拥有多少参数」和「每步用多少参数」是可以分离的。Dense 模型把这两个数字绑死了；MoE 把它们解耦了。

需要特别指出的是 MoE 的两个代价：其一，所有专家的参数必须同时驻留在 GPU 显存里，因为任何 token 都可能路由到任何专家；其二，在多 GPU 分布式部署中，当 token 被路由到另一块 GPU 上的专家时，需要跨设备通信，引入额外延迟。这两个代价在工程落地时至关重要，是选择 Dense 还是 MoE 的核心权衡点。

  • Transformer 每层的 FFN 替换为 $E$ 个并行专家；Attention 层不变，各层共享
  • 门控得分 $G_i(x) = \mathrm{Softmax}(\mathrm{TopK}(xW_g))_i$，仅被选中的 $k$ 个非零

典型配置：
$E = 8,\ k = 2$（Mixtral）
$E = 64,\ k = 6$（DeepSeek-V2 路由专家）
$E = 2048,\ k = 2$（Switch Transformer 极端版）

MoE 的修改只发生在 FFN 层，Attention 部分完全不变。路由器是一个小型线性层，把 token 表示 $x$ 映射到 $E$ 维得分向量，取 top-k 个正分值后用 softmax 归一化。

top-k 操作本身不可微，但实践中通常直接用 straight-through 估计，或者依赖辅助损失提供足够的训练信号。

一个常被忽视的细节：从整个 batch 的视角看，所有专家在每一步都在工作（不同 token 分流给不同专家），所以硬件利用率并不低；但对每个 token 而言，它只「看到」了 $k$ 个专家。这种全局并行、局部稀疏的特性，是 MoE 能保持高吞吐量的关键。

06

负载均衡挑战

Load Balancing
  • 若不加约束，路由器初期随机偏差 → 某几个专家被正反馈循环强化 → 专家崩塌（Expert Collapse）
  • 崩塌后：头部 1-2 个专家承接几乎所有 token，其余专家几乎从不更新，大量参数浪费
  • 解决①：辅助均衡损失 $\mathcal{L}_{\mathrm{aux}}$，惩罚负载集中；建议 $\alpha \approx 10^{-2}$
  • 解决②：Token Capacity 硬上限——每专家每 batch 最多处理 $\lfloor T/E \cdot C \rfloor$ 个 token，超额 token 跳过（$C$ 为容量因子，通常 1.0–1.25）
$$\mathcal{L}_{\mathrm{aux}} = \alpha \sum_{i=1}^{E} \underbrace{f_i}_{\text{实际比例}} \cdot \underbrace{P_i}_{\text{路由概率均值}}$$
专家崩塌的正反馈
专家 $i$ 初始偏强
↓
更多 token 路由到 $i$
↓
$i$ 获得更多梯度更新
↓
$i$ 变得更强 → 循环
$f_i \cdot P_i$ 最小化等价于让两个分布尽量均匀；二者之积在均匀分布时最小

专家崩塌是 MoE 训练中最常见的失败模式。核心原因是路由器的梯度流存在正反馈：被选中的专家得到更新，更新后更强，于是被更多地选中。辅助损失通过对 $f_i \cdot P_i$ 的求和惩罚来打破这个循环。

Token Capacity 机制则是一个硬约束，当某个专家已满时，超额 token 要么被丢弃，要么路由给 reserve expert（备用专家）。这在推理时确保所有设备的负载更均衡，但在训练时需要小心不要因为频繁丢弃 token 而影响学习效果。DeepSeek 使用的是更细粒度的 device-level 均衡，而不是单纯的 global 均衡。

Mixtral 8×7B 的架构有几个值得关注的细节：首先，它使用了 GQA（8 个 KV 头），这对推理时的 KV cache 大小有直接影响——从原来的 32 个 KV 头降到 8 个，缓存减少 75%。其次，每个专家的 FFN 维度 14336 相当大，这是为了确保每个专家都有足够的表达能力。

基准数据非常直观：Mixtral 8×7B 用 Llama 2 13B 的推理成本（约 13B 激活参数），在 MMLU 和 HumanEval 上超过了需要 70B 计算量的 Llama 2 70B。特别是 HumanEval（代码生成基准），Mixtral 的 40.2 远高于 Llama 2 70B 的 29.9，说明 MoE 的专家分化可能对代码这类结构性强的任务特别有帮助——这也呼应了后面假说一的讨论。

长上下文技术

Long Context Techniques

我们进入第二个主题：长上下文技术。MoE 解决「如何在有限算力下塞入更多知识」，长上下文解决「如何在单次推理中处理更多信息」。接下来三张幻灯片分别讲瓶颈、位置编码、以及高效注意力方案。

09

注意力机制的扩展瓶颈

Why Attention Doesn't Scale
  • 标准自注意力的时间与显存复杂度均为 $O(n^2)$，$n$ 为序列长度
  • 核心步骤 $QK^\top \in \mathbb{R}^{n\times n}$ 需要完整落盘，无法逐行流式处理
  • 因果掩码只使下三角非零，但计算仍需完整矩阵（PyTorch 标准实现）
  • 上下文需求：GPT-4 128K → Claude 3 200K → Gemini 1.5 1M
$$\mathrm{Attention}(Q,K,V) = \mathrm{softmax}\!\left(\frac{QK^\top}{\sqrt{d_k}}\right)V$$
序列每增长 $10\times$，时间和显存成本均增长 $\mathbf{100\times}$。
单层 $QK^\top$ 显存（fp16）
序列长度 $n$矩阵大小32 层累计
4K32 MB~1 GB
32K2 GB~64 GB
128K32 GB~1 TB ✗
1M2 TB~64 TB ✗✗
推理时还需 KV cache：每 token 需存 $2 \times L \times n_h \times d_h$ 浮点数；32K ctx、32 层、128 heads、128 dim ≈ 2 GB

$O(n^2)$ 问题在小上下文时几乎感觉不到，但随着 n 增大会指数级爆炸。右侧表格的数字说明问题：128K 上下文下仅注意力矩阵就需要 1 TB 显存，这在任何现有 GPU 上都是不可能存在的。

推理时还有额外的 KV cache 问题：自回归生成时，每新生成一个 token，需要把它对应的 K 和 V 缓存下来以备后续 token 使用。这是一个只增不减的缓存，上下文越长，缓存越大。128K 上下文、32 层、128 heads、head dim 128 的配置下，KV cache 约 2 GB，这占了相当大一块显存预算。

这两个问题——计算时的 $n^2$ 矩阵和推理时的线性 KV cache——是促使 FlashAttention、GQA、MLA 等技术出现的直接动力。

旋转位置编码（RoPE）

Rotary Position Embedding
  • Transformer 排列不变，必须额外编码顺序信息；原始正弦编码（Vaswani 2017）在训练外长度泛化性差
  • RoPE（Su et al., 2021）：给 $Q$/$K$ 的每对维度 $(d_{2i}, d_{2i+1})$ 施加旋转角 $m\theta_i$，$\theta_i = 10000^{-2i/d}$
  • 关键性质：$q_m^\top k_n$ 只依赖相对位置 $n-m$，绝对位置互相抵消
旋转矩阵结构（每对维度）
$$R_\theta^m = \begin{pmatrix} \cos m\theta_i & -\sin m\theta_i \\ \sin m\theta_i & \cos m\theta_i \end{pmatrix}$$
$q_{2i}, q_{2i+1}$
×
$R_\theta^m$
=
$\tilde{q}_m$
$k_{2i}, k_{2i+1}$
×
$R_\theta^n$
=
$\tilde{k}_n$
相对位置性质
$$\tilde{q}_m^\top \tilde{k}_n = q^\top R_\theta^{n-m} k$$
得分 ∝ 相对位置 $(n-m)$，绝对值不重要

YaRN / RoPE Scaling：训练时用短上下文，推理时将 $\theta_i$ 等比缩放 → 无需重新训练即可扩展上下文长度（Llama 3 采用）

RoPE 的旋转矩阵是块对角的，每对相邻维度独立旋转。低维度的 $\theta_i$ 较大（旋转快，编码短距离），高维度的 $\theta_i$ 较小（旋转慢，编码长距离），这类似于傅里叶变换中不同频率的正弦波。

相对位置性质是 RoPE 最核心的数学优势：当计算注意力得分 $\tilde{q}_m^\top \tilde{k}_n$ 时，两个旋转矩阵的乘积 $R^m \cdot (R^n)^\top = R^{n-m}$ 只依赖相对位置差。这意味着即使推理时出现了训练时没见过的绝对位置，相对距离关系仍然保持正确。

YaRN 等上下文扩展方法的思路是：把 $\theta_i$ 乘以一个小于 1 的缩放因子，让旋转速度变慢，等效于「拉伸」了位置编码的动态范围，使得更长的相对距离也能被覆盖。这样在原始短上下文训练的模型上，只需少量长文本微调甚至无需微调就能显著扩展上下文。

这张表列出了五种主要方案，每种解决问题的角度不同。FlashAttention 的贡献纯属工程层面：把 GPU SRAM（L1/L2 cache）利用起来做分块计算，避免大矩阵写回 HBM（全局显存），从而把显存复杂度从 $O(n^2)$ 降到 $O(n)$，同时速度提升 2-4 倍。FlashAttention v2/v3 进一步优化了并行化，目前已成为所有主流框架的默认实现。

GQA 的直觉：在推理时，KV cache 是显存的大头，且 KV cache 不需要每个 Query 头都独立——实验发现，让多个 Q 头共享一对 KV 头，性能几乎不下降，而 KV cache 大小可以压缩 $g$ 倍。MQA（$g=1$ 极端情况）压缩最多，但有时精度损失明显，所以 GQA 成为更常见的折中方案。

滑动窗口注意力适合不需要精确全局依赖的任务，Mistral 7B 是代表性实现，它在不同层混合使用全注意力和滑动窗口注意力来兼顾局部和全局信息。

MoE 与
长上下文的结合

When MoE Meets Long Context

现在把两者放在一起。MoE 解决参数效率，长上下文解决信息容量，但组合不是简单叠加——需要面对三类新的工程挑战。DeepSeek-V2 是目前最完整的解答之一。

14

结合时的三重挑战

Three Tensions
挑战 01

KV 缓存膨胀

KV cache ∝ $n \times L \times 2n_h d_h$，与 MoE 无关但随序列线性爆炸。128K ctx、32 层、128 heads、dim 128 ≈ 128 GB（fp16），超出单 GPU 预算。

解法：GQA 降低头数；MLA 压缩潜在空间；量化（fp8/int4）
挑战 02

路由独立性

每 token 独立路由，长序列中语义连续的相邻 token 可能分发到截然不同的专家。跨越远距离的语义连贯性完全依赖 Attention，MoE 路由无法辅助。

解法：共享专家（DeepSeek）为所有 token 提供通用基础处理
挑战 03

All-to-All 通信

分布式训练中专家分散在不同节点；每层均需 all-to-all 发送/接收 token。假设 512 token、$d=4096$、bf16：单层通信量 ≈ $512 \times 4096 \times 2 \approx$ 4 MB，32 层连续 → 显著增加端到端延迟。

解法：Expert Parallel + Pipeline Parallel；通信与计算 overlap
核心：MoE 扩大参数容量，长上下文扩大信息容量，但二者叠加的显存与通信压力倍增，需要专门的架构创新来抵消

三个挑战各有对应的工程解法，但没有一个是「免费」的。KV cache 膨胀可以用 GQA 或 MLA 缓解，但 GQA 引入了轻微精度损失，MLA 引入了额外的上采样计算；共享专家在一定程度上保证通用语义，但增加了每 token 的激活参数量；通信 overlap 需要精细的调度，增加了系统实现复杂度。

DeepSeek-V2 的价值在于它同时处理了这三个问题：MLA 解决 KV cache，共享专家 + 细粒度 MoE 解决路由稳定性，设备级均衡损失 + EP 并行解决通信效率。

案例：DeepSeek-V2

MoE + Long Context in Practice
  • 总参数 236B，激活 21B（2 共享 + 6 路由 top-6/160），上下文 128K
  • 细粒度 MoE：将每个专家拆分更小（$d_{\text{ffn}}$ 减半），专家数翻倍 → 路由粒度更细，分配更灵活
  • MLA（Multi-head Latent Attention）：将所有 KV 头压缩到一个低秩向量 $c_{KV} \in \mathbb{R}^{d_c}$ 缓存，解码时上采样还原 K/V
  • 训练成本约为同规模 Dense 的 $\tfrac{1}{10}$，性能与 GPT-4 Turbo 相当

DeepSeek-V2 的两个核心创新值得详细讲解。

第一是细粒度 MoE：把原来大专家拆成更小的专家（FFN 中间维度减半），同时专家数量翻倍，然后仍然激活相同比例的专家。这样做的好处是每个专家更「专」，路由决策的粒度更细，token 可以在更多专家中找到最合适的组合。实验表明，细粒度配置在相同激活参数量下性能更好。

第二是 MLA。传统 KV cache 把每层每个 Attention 头的 Key 和 Value 都存下来，显存需求随头数线性增长。MLA 的思路是：先把 hidden state 压缩成一个低秩向量 $c_{KV}$（维度 512），然后推理时动态解压缩出 K 和 V。缓存 $c_{KV}$ 而不是 K/V，把缓存大小从 ~576 维压到 ~38 维（按幻灯片的配置估算），节省 93%。

这里有一个工程技巧：在推理部署时，可以把解压缩矩阵 $W_{UK}$/$W_{UV}$ 提前与后续的 Query 投影矩阵合并（矩阵吸收），消除额外的上采样计算开销，使 MLA 的推理延迟不高于标准 GQA。

为什么中间容易失忆？
训练时短文本占多数 → 模型注意力「习惯于」偏向开头与结尾；中间段的注意力权重在归一化后被稀释。这就是「中间迷失」（Lost in the Middle）现象。

「大海捞针」测试由 Greg Kamradt 在 2023 年提出，现已成为评测长上下文的标准 benchmark 之一，完整版测试会在不同文档位置和不同文档长度下系统评测召回率，生成一张热力图。

我们的简化版有同样的直觉价值。右侧表格给出了预期结果：强模型（Claude 3.5、GPT-4o）在所有位置表现良好；较弱的模型在中间位置会出现明显的遗忘现象。这验证了 Liu et al. (2023) 的 Lost-in-the-Middle 发现。

Prompt 里加了一句「不要说文章没有提到」，这是为了阻止模型在找不到答案时撒谎地回答「未找到相关信息」——这种提示工程技巧在实际评测时很常见。现在大家来实际操作一下。（等待同学操作约 1-2 分钟）

关键假说

Critical Hypotheses

最后一部分，我提出三个目前研究界尚未给出清晰答案的假说。每个假说我会说明现有证据，以及为什么它仍然值得追问。

19

三个值得追问的假说

Open Questions
假说 01

专家真的「专业化」了吗？

探针实验（Zoph et al. 2022；Lepikhin et al. GShard）发现部分专家对特定语言或句法结构有统计偏好，如代码/数学专家。但这是真正的语义分工，还是负载均衡的副产物？

  • · 路由决策对输入的微小扰动高度敏感
  • · 随机权重初始化的不同 seed 会产生不同的专家分配模式
若专业化是假象，MoE 的「可解释性」优势是否不存在？
假说 02

长上下文真的被用到了吗？

Liu et al. (2023) TACL 论文：模型在多文档 QA 任务中，对开头和结尾的信息利用率最高，中间段信息利用率显著下降（U 形曲线）。

开头←召回率→结尾

即使 Gemini 1.5 扩展到 1M，中间段利用率改善有限；RAG 可能比「更长上下文」更实用。

「长上下文」 ≠「有效利用长上下文」
假说 03

MoE + 长上下文是否相互增强？

两者动机独立：MoE 解决参数效率，长上下文解决信息容量。Gemini 1.5 将二者结合并在 1M 上下文 NIAH 测试中表现优异，但：

  • · 消融实验中「MoE 专为长上下文训练」的增益尚不明确
  • · 系统复杂度（KV cache + all-to-all）随两者叠加倍增
  • · 新范式（Mamba/SSM）可能以线性复杂度统一解决二者
是协同涌现，还是两个独立方案的工程叠加？

假说一的核心争议在于因果性。相关性实验（某专家被特定类型 token 激活的频率更高）不等于专家真正学到了独立的语义能力。如果随机种子改变后，哪个专家负责哪类 token 就整个打乱了，那「专业化」更可能是一种统计涌现而非设计结果。这对 MoE 的可解释性主张是个挑战。

假说二的关键数据点是 Liu et al. 2023 的实验：在 20 个文档中找到含答案的文档，当该文档在上下文中部时，模型准确率下降 10-20 个百分点，即使模型的上下文窗口远大于文档总长度。这说明问题不在于上下文是否够长，而在于注意力机制的归纳偏置导致中间位置的权重被系统性稀释。

假说三是最开放的一个。目前没有强力的消融实验证明 MoE 和长上下文的组合比「等参数量的 Dense 长上下文模型 + 等参数量的 Dense 短上下文 MoE 模型」更好。也许未来架构（如 SSM 或新型 linear attention）会提供一个更统一的解决方案，而不是在 Transformer 上打各种补丁。这是一个值得研究生朋友们长期关注的方向。
