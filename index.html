大家好，我是 StarryPurple。今天的报告题目是《混合专家模型与长上下文》。

大语言模型的发展有两条清晰的主线。第一条是参数规模的急剧扩张——从 2019 年 GPT-2 的 15 亿参数，到如今 GPT-4 估计超过万亿参数，五年增长了近千倍。第二条是上下文窗口的持续拉长——从最初的 2K token，到 Gemini 1.5 的百万 token，同样增长了约千倍。

这两条主线背后各有一个核心矛盾：参数越多，推理越贵；上下文越长，注意力计算越慢。MoE（混合专家模型）给出了第一个矛盾的优雅解法——让模型「拥有」很多参数，但每次只「使用」其中一小部分。长上下文技术则通过位置编码扩展、高效注意力、稀疏注意力等手段，解决第二个矛盾。

今天的报告大约四十五分钟，分六个部分。我们先看背景，再分别深入 MoE 和长上下文，然后看两者如何结合，接着用 Claude 做几个实测验证，最后总结和展望。

【1:30 – 5:00 · 规模定律】

我们从规模定律开始。2020 年 Kaplan 等人在 OpenAI 发表了一篇影响深远的论文，核心结论是：模型性能（用交叉熵损失衡量）随参数量 N、数据量 D、算力 C 呈幂律增长，而且在当时的实验范围内看不到明显的上限。这意味着，只要你愿意投入更多资源，模型就会持续变好——这是整个大模型军备竞赛的理论基础。

但 2022 年 DeepMind 的 Chinchilla 论文修正了一个重要细节：在固定算力预算下，参数量和训练 token 数应该同步扩大，最优比例大约是 1:20。也就是说，一个 70B 的模型应该用 1.4T token 来训练。Chinchilla 用 70B 参数、1.4T token 训练，在 MMLU 上达到 67.5%，超过了用 280B 参数但只训练了 300B token 的 Gopher 的 60%。右边的表格展示了不同算力预算下的最优配置，大家可以看到 Llama 2 和 Llama 3 基本遵循了这个比例。

但无论怎么优化训练，推理成本有一个硬约束：每生成一个 token，需要大约 2N 次浮点运算，N 是参数量。ChatGPT 峰值时据估计需要约 3 万张 A100，月均成本超过 5000 万美元。

底部的对比展示了核心矛盾：以 70B Dense 模型为例，日均 1000 万请求的月成本约 75 万美元。但如果换成 MoE 架构——总参数 280B，每次只激活 70B——推理成本完全相同，但模型的知识容量扩大了四倍。这就是 MoE 的核心价值主张：用 Dense 级别的推理成本，撬动远大于 Dense 的参数容量。

04

稠密 vs 稀疏激活

Dense vs Sparse Activation

Dense（稠密）

  • 每个 token 激活全部参数
  • 推理 FLOPs ∝ 总参数量 $N$
  • 硬件利用率高，内存访问模式规整
  • 代表：GPT-4（~1T est.）、Llama 3（405B）
$$\text{FLOPs} \approx 2 \cdot N \cdot T$$

Sparse MoE

  • 每个 token 仅激活 $k$ 个专家（$k \ll E$）
  • 推理 FLOPs ∝ $N_{\text{act}} \ll N$
  • 代价：所有专家须全驻显存；分布式需额外通信
  • 代表：Mixtral 8×7B（47B→13B act）、DeepSeek-V2
$$\text{FLOPs} \approx 2 \cdot N_{\text{act}} \cdot T \quad (N_{\text{act}} \ll N)$$
以 Dense 级别的推理 FLOPs，撬动远大于 Dense 的参数容量——参数量 ≠ 激活参数量。这是 MoE 的核心价值主张。

【5:00 – 7:00 · 稠密 vs 稀疏】

这张幻灯片把核心区分讲清楚：「拥有多少参数」和「每步用多少参数」是可以分离的。Dense 模型把这两个数字绑死——GPT-4 估计有上万亿参数，每生成一个 token 就要全部激活一遍。Llama 3 的 405B 版本也是如此。

MoE 把它们解耦了。以 Mixtral 8×7B 为例，总参数 47B，但每个 token 只激活 2 个专家，实际计算量只有 13B 参数对应的 FLOPs。DeepSeek-V2 更极端：236B 总参数，只激活 21B。

但 MoE 有两个不可忽视的代价。第一，所有专家的参数必须同时驻留在显存中——你不能只加载被激活的专家，因为每个 token 的路由决策是动态的，你事先不知道哪些专家会被选中。第二，在分布式部署中，不同专家分布在不同 GPU 上，token 路由到远端专家时需要跨设备通信，这会引入额外延迟。这两个代价在工程落地时至关重要，我们后面讨论 DeepSeek-V2 时会看到它们如何被缓解。

底部的总结框是今天的核心命题：以 Dense 级别的推理 FLOPs，撬动远大于 Dense 的参数容量。参数量不等于激活参数量——这是 MoE 的全部精髓。

【7:15 – 10:30 · MoE 核心机制】

MoE 的修改只发生在 Transformer 的 FFN 层，Attention 部分完全不变。顶部的流程图展示了完整的路由过程：输入 token 的隐藏状态 x 经过一个小型线性层（路由器），映射到 E 维得分向量，取 top-k 后用 softmax 归一化，得到每个被选中专家的权重。最终输出是被选中专家输出的加权求和。

这里有一个关键的视角切换：从单个 token 的角度看，它只「看到」了 k 个专家，计算量是原始 FFN 的 k/E；但从整个 batch 的角度看，所有专家在每一步都在工作——不同 token 被分流给不同专家。所以 MoE 的硬件利用率并不低，它是全局并行、局部稀疏的。

右下角列出了三种典型配置。Mixtral 用 8 个专家取 top-2，这是最简单的配置。DeepSeek-V2 用 160 个路由专家取 top-6，专家数量大幅增加但每个专家更小。Switch Transformer 走向另一个极端——2048 个专家但每个 token 只选 1 个。

最后提一下 Noisy Top-K：Shazeer 在 2017 年的原始论文中引入了可学习的噪声项，在路由得分上加入高斯噪声，鼓励路由多样性，防止所有 token 都涌向同一个专家。这是解决负载均衡问题的早期尝试，下一张幻灯片我们会看到更系统的方案。

负载均衡挑战

Load Balancing
  • 若不加约束，路由器初期随机偏差 → 正反馈循环 → 专家崩塌（Expert Collapse）
  • 解决①：辅助均衡损失 $\mathcal{L}_{\mathrm{aux}}$，惩罚负载集中；$\alpha \approx 10^{-2}$
  • 解决②：Token Capacity 硬上限（Switch Transformers）——超额 token 溢出跳过
    $\text{Expert Capacity} = \lfloor T/E \rfloor \times C,\quad C \in [1.0,\,1.25]$
  • 解决③：Router z-loss（ST-MoE, Zoph 2022）——惩罚 logits 绝对值过大，减少数值误差
    $\mathcal{L}_z = \tfrac{\alpha_z}{B}\sum_i\!\left(\log\textstyle\sum_j e^{x_{ij}}\right)^{\!2}$
$\mathcal{L}_{\mathrm{aux}} = \alpha \sum_{i=1}^{E} \underbrace{f_i}_{\text{实际比例}} \cdot \underbrace{P_i}_{\text{路由概率均值}}$
专家崩塌的正反馈
专家 $i$ 初始偏强
↓
更多 token 路由到 $i$
↓
$i$ 获得更多梯度更新
↓
$i$ 变得更强 → 循环
$f_i \cdot P_i$ 最小化等价于让两个分布尽量均匀；二者之积在均匀分布时最小

【10:30 – 13:00 · 负载均衡】

负载均衡是 MoE 训练中最常见的失败模式，值得花点时间讲清楚。右侧的流程图展示了「专家崩塌」的正反馈循环：假设训练初期某个专家 i 因为随机初始化恰好稍强一点，那么路由器会把更多 token 分配给它；更多 token 意味着更多梯度更新，专家 i 变得更强；更强又吸引更多 token——如此循环，最终只有少数专家被使用，其余专家「饿死」。这就是专家崩塌。

解决方案有三种。第一种是辅助均衡损失 L_aux：对每个专家，计算它实际接收的 token 比例 f_i 和路由概率均值 P_i，然后惩罚 f_i 乘以 P_i 的总和。数学上可以证明，当所有专家的负载完全均匀时，这个乘积之和最小。系数 α 通常取 0.01 左右——太大会干扰主任务学习，太小则无法阻止崩塌。

第二种是 Token Capacity 硬上限，由 Switch Transformer 提出：每个专家设定一个容量上限，超额的 token 直接溢出，要么跳过 MoE 层，要么路由给备用专家。容量因子 C 通常在 1.0 到 1.25 之间。

第三种是 Router z-loss，由 ST-MoE 论文提出：惩罚路由器 logits 的绝对值过大。这看起来和负载均衡无关，但实际上 logits 过大会导致 softmax 中的 exp 溢出，产生数值误差，进而导致路由决策不稳定。ST-MoE 论文证明加入 z-loss 后训练稳定性显著提升。

这三种方法在实践中通常组合使用。但辅助损失有一个根本性的问题：它的梯度会干扰主任务的学习。DeepSeek-V3 后来提出了一种完全不同的方案，我们稍后会讲到。

【13:00 – 15:00 · 代表性工作】

这张时间线梳理了 MoE 从 1991 年到 2024 年的演进。MoE 的概念最早由 Jacobs 等人在 1991 年提出，但当时还没有 Transformer，应用场景完全不同。真正进入大模型时代是 2017 年 Shazeer 在 Google 的工作——他把 MoE 层插入到 LSTM 中，用 2048 个专家和 Noisy Top-K 门控，训练了 137B 参数的模型。

2020 年 GShard 把 MoE 扩展到 600B 参数，引入了 Top-2 随机路由和 Expert Capacity 机制。2022 年 Switch Transformer 走向另一个极端——每个 token 只选 1 个专家，大幅简化了路由逻辑，但需要更大的 Capacity Factor 来补偿。Switch Transformer 把模型扩展到了 1.6T 参数。

2024 年是 MoE 的爆发年。Mixtral 8×7B 是第一个广泛使用的开源 MoE 模型，证明了 MoE 在实际部署中的可行性——47B 总参数，只激活 13B，性能接近 Llama 2 70B。DeepSeek-V2 和 V3 则代表了最新的架构演进。

表格底部的趋势总结很重要：专家从「少而大」（Mixtral 的 8 个大专家）演变为「多而小」（DeepSeek 的 256 个细粒度专家）；路由从 Top-2 扩展到 Top-6 甚至 Top-8；共享专家从无到有，成为标配。下一张幻灯片我们详细看 DeepSeek 的架构。

DeepSeek MoE 架构详解

Fine-Grained Experts & Shared Experts
  • 细粒度专家：将标准 FFN 拆分为更小的专家（$d_{\text{ffn}}$ 减半，专家数翻倍），路由粒度更细，分配更灵活
  • 共享专家（$N_s$ 个）：所有 token 必经，提供通用语义基础；路由专家负责差异化处理
  • Group-Limited Greedy TopK：将专家分组，每组内独立选 top-$k_g$，防止 token 集中在少数组
  • DeepSeek-V2：160 路由 + 2 共享，top-6；V3：256 路由 + 1 共享，top-8
$y = \sum_{s=1}^{N_s} \mathrm{FFN}_s(x) + \sum_{i \in \mathrm{TopK}} G_i(x) \cdot \mathrm{FFN}_i(x)$

【15:00 – 18:30 · DeepSeek MoE 架构】

DeepSeek MoE 有两个核心创新，我分别讲。

第一个是细粒度专家。传统 MoE 如 Mixtral，每个专家就是一个完整的 FFN，中间维度 14336。DeepSeek 把每个专家的 FFN 中间维度减半到 1536，但专家数量从 8 个增加到 160 个。总参数量不变，但路由粒度更细了——模型可以更精确地为每个 token 选择最合适的专家组合。右侧的架构对比图直观展示了这个区别：Mixtral 是 8 个大方块选 2 个，DeepSeek 是 160 个小方块选 6 个。实验表明，在相同激活参数量下，细粒度配置的性能更好。

第二个是共享专家。在标准 MoE 中，每个 token 只经过被路由选中的专家，不同 token 可能激活完全不同的专家组合。这带来一个问题：语义连贯性完全依赖 Attention 层，FFN 层没有任何「公共基础」。DeepSeek 引入了 N_s 个共享专家（V2 中是 2 个），所有 token 都必须经过共享专家处理，获得通用语义基础；路由专家只负责差异化的专业处理。公式中可以看到，输出是共享专家输出加上路由专家的加权输出。

左下角的框介绍了 V3 的一个重要突破：移除辅助均衡损失。前面讲过，辅助损失虽然能防止崩塌，但它的梯度会干扰主任务学习——这是一个根本性的矛盾。V3 的解法非常巧妙：给每个专家维护一个 bias 项，训练时动态调节——如果某个专家负载不足，就给它加正 bias 吸引更多 token；负载过高就减小 bias。关键是：推理时完全不使用这个 bias，所以它不会影响模型的输出质量。这比辅助损失优雅得多。

长上下文面临三个核心挑战，对应顶部的三张卡片。

第一个是 Prefill 阶段的 O(n²) 计算。标准注意力需要计算 Q 乘以 K 转置，得到一个 n×n 的矩阵。序列长度增加 10 倍，计算量和显存都增加 100 倍。右边的表格很直观：4K 序列时单层注意力矩阵只需 32MB，32 层累计约 1GB，完全可以放进一张 GPU；但 128K 时单层就需要 32GB，32 层累计 1TB——这已经超出了任何单张 GPU 的显存。到 1M 时更是天文数字。

第二个是 KV Cache 膨胀。自回归生成时，每生成一个新 token，都要把它的 K 和 V 向量缓存起来，供后续 token 使用。这个缓存只增不减，大小正比于序列长度乘以层数乘以注意力头数。32K 上下文、32 层、128 个头的配置下，KV cache 就需要约 2GB。

第三个是 Attention Distractions，也叫注意力稀释。当序列超过百万 token 时，注意力矩阵中绝大多数权重接近零，但 softmax 归一化使这些近零元素在矩阵乘法中不断积累，关键信号被无关内容「分心」。这不是数值精度问题，而是信息论层面的问题。

这三个问题分别催生了不同的技术方案：FlashAttention 解决显存问题、GQA 和 MLA 解决 KV cache 问题、稀疏注意力解决 Attention Distractions 问题。接下来我们逐一展开。

【21:30 – 24:00 · RoPE 与位置编码扩展】

在讲高效注意力之前，先解决一个前置问题：位置编码。Transformer 本身是置换不变的，需要位置编码告诉模型 token 的顺序。RoPE 是目前最主流的方案，Llama、Mistral、DeepSeek 都在用。

RoPE 的核心思想是：给 Q 和 K 的每对相邻维度施加一个旋转角 mθ_i，其中 m 是位置，θ_i 是频率参数。旋转后的 Q 和 K 做内积时，结果只依赖相对位置差 n-m，而不是绝对位置。这个相对位置性质是 RoPE 的核心数学优势。

频率参数 θ_i 的设计很巧妙：低维度的 θ_i 大，旋转快，编码短距离关系；高维度的 θ_i 小，旋转慢，编码长距离关系。这类似于傅里叶变换中的多频率分解。

但 RoPE 有一个问题：模型在 8K 上下文上训练，直接推理 128K 时性能会急剧下降，因为模型从未见过那么大的旋转角。右侧的表格列出了四种上下文扩展方法。PI（Position Interpolation）最简单——直接线性缩放 θ，把 128K 的位置映射到 8K 的范围内。NTK-aware 更聪明——它认识到高频维度编码短距离关系，不需要缩放，只缩放低频维度。YaRN 在 NTK 基础上加入注意力温度修正。LongRoPE 采用两阶段渐进式扩展。

实际应用中，Llama 3 采用 YaRN 从 8K 扩展到 128K；LongRoPE 声称可以将 4K 模型扩展到 2048K，但实际效果还有待验证。

【24:00 – 26:30 · 高效注意力】

这张幻灯片对比了五种注意力实现方式。左侧的表格是核心信息。

标准注意力的时间和显存都是 O(n²)，这是我们要优化的基线。FlashAttention 的贡献纯属工程层面：它把 GPU 的 SRAM（片上高速缓存）利用起来做分块计算，避免把巨大的 n×n 注意力矩阵写回 HBM（显存）。时间复杂度没变，还是 O(n²)，但显存从 O(n²) 降到了 O(n)，实际速度提升 2-4 倍。FlashAttention v2 和 v3 进一步优化了并行化策略。现在所有主流框架——PyTorch、JAX、Triton——都默认使用 FlashAttention。

滑动窗口注意力（SWA）是一种近似方法：每个 token 只关注窗口内的 w 个邻居。时间和显存都降到 O(nw)，但丢失了长距离依赖。右侧的注意力模式图直观展示了区别：标准注意力是全蓝的方阵，滑动窗口是一条对角带。

GQA（Grouped Query Attention）解决的是 KV cache 问题。推理时 KV cache 是显存大头，GQA 让每 g 个 Q 头共享一对 KV 头，KV cache 压缩 g 倍，性能几乎不下降。MQA 是极端情况——所有 Q 头共享 1 对 KV 头。GQA 是更常见的折中，Llama 3 和 Mistral 都在用，g 大约取 H/4。

底部的总结：FlashAttention 已成为默认实现，GQA 已成为标配。这两个技术是当前所有长上下文模型的基础设施。

【26:30 – 29:00 · 线性注意力与混合架构】

FlashAttention 优化了常数因子，但没有改变 O(n²) 的本质。线性注意力试图从根本上解决这个问题。

核心思想是改变计算顺序。标准注意力先算 QK^T（得到 n×n 矩阵），再乘 V；线性注意力用特征映射 φ 替代 softmax，先算 φ(K)^T V（得到 d×d 矩阵），再左乘 φ(Q)。当 d 远小于 n 时——也就是长序列场景——复杂度从 O(n²d) 降到 O(nd²)。

更重要的是，线性注意力的历史状态可以压缩为一个 d×d 的矩阵，不需要存储完整的 KV cache。这意味着推理时的显存占用与序列长度无关，天然适合超长序列。

但线性注意力有一个明显的缺点：精确信息检索能力不如 softmax 注意力。比如在 Needle-in-a-Haystack 测试中，纯线性注意力的表现明显更差。

MiniMax-01 给出了一个优雅的折中方案：混合架构。每 8 层中 7 层用 Lightning Attention（线性），1 层用标准 Softmax Attention。右侧的示意图展示了这个 7:1 的比例。少量 Softmax 层保证了精确的信息检索能力，大量线性层降低了整体计算和显存开销。右下角的表格显示，混合架构在 NIAH 精度上接近纯 Softmax，但时间复杂度接近线性。MiniMax-01 在 4M token 上下文下仍能高效运行。

【29:00 – 31:30 · 稀疏注意力】

这张幻灯片介绍 2024-2025 年稀疏注意力的三个最新进展，它们都受到了 MoE 稀疏激活思想的启发——这也是今天报告把 MoE 和长上下文放在一起讲的原因之一。

MoBA（Mixture of Block Attention）的思路最直接：把序列分成若干块，为每个 Query 计算它与各块平均 Key 的相似度作为路由得分，选 top-k 个最相关的块来做注意力。这和 MoE 的路由机制几乎一模一样——只不过 MoE 是在专家空间做稀疏选择，MoBA 是在序列空间做稀疏选择。右下角的注意力模式图展示了 MoBA 的效果：不同 Query 关注不同的块，形成不规则的稀疏模式。

NSA（Native Sparse Attention）更复杂，同时维护三条路径：Token Compression 用 MLP 把整块 KV 压缩为单一 KV，提供全局粗粒度信息；Token Selection 选全局最重要的 top-n 个 token，提供精确的远距离信息；Sliding Window 保留局部窗口近邻信息。三条路径通过可学习的门控融合。NSA 的注意力模式图中，浅蓝色是压缩的全局信息，深蓝色是选择的重要 token 和滑动窗口。

MInference 最轻量——它不修改模型权重，只在推理时为每个注意力头识别最重要的 4K 个 Key 列，跳过其余列。在 128K 上下文下，这 4K 列就能召回超过 96% 的注意力得分。

底部的总结框点明了核心联系：MoBA 的思路与 MoE 一脉相承——每个计算单元只处理最相关的子集。从参数空间的稀疏激活，迁移到注意力空间的稀疏计算。

【31:30 – 33:30 · 推测解码】

推测解码是长上下文推理加速的另一个重要方向。经典推测解码的思路是：用一个小模型（drafter）快速生成一串候选 token，然后大模型一次性验证这些候选。如果候选正确，就省去了大模型逐个生成的时间；如果不正确，从第一个错误位置重新开始。在短上下文下，这可以带来 2-3 倍加速。

但长上下文下有一个问题：drafter 本身也受 O(n²) 制约。当上下文达到 128K 时，drafter 也变慢了，加速比急剧下降。

左侧介绍了三种针对长上下文的改进方案。RAPID 的巧妙之处在于：让 drafter 不看全文，只看 RAG 检索到的相关片段。这样 drafter 的上下文从 128K 压缩到约 4K，速度恢复了。验证仍由大模型在完整上下文上进行，所以输出质量不受影响。

SuffixDecoding 利用了长文档中的重复模式：用后缀树索引上下文中的 n-gram，当生成的 token 序列匹配已有模式时，直接复制后续 token。这在代码生成和法律文本等重复性强的场景下特别有效，加速比可达 3-5 倍。

Speculative Prefill 针对的是首 token 延迟（TTFT）——长上下文下 prefill 阶段是主要延迟来源。先用小模型快速扫描全文，识别重要 token 位置，大模型只对这些位置做精确计算。TTFT 可降低 2-5 倍。

右下角的总结很重要：这三种方法可以正交组合——Speculative Prefill 加速首 token，RAPID 加速后续生成。

【33:30 – 35:30 · 上下文窗口演进】

这张时间线把前面讲的所有技术串联起来。从 2019 年 GPT-2 的 1K token，到 2024 年 Gemini 1.5 的 1M token，五年增长了 1000 倍。

时间线底部标注了每个里程碑背后的关键技术。GPT-2 和 GPT-3 用的是正弦位置编码，上下文窗口受限于训练时的最大长度。2022 年 FlashAttention 的出现是第一个转折点——它让 16K 上下文在工程上变得可行。2023-2024 年 RoPE + GQA + YaRN 的组合把窗口推到了 128K。Gemini 1.5 达到 1M，背后是 MoE + 滑动窗口注意力的组合。

底部的两张卡片介绍了训练策略。渐进式扩展已成为标准做法：先在短上下文（4K-8K）上充分预训练，然后逐步扩展到 32K、128K、1M，每个阶段用少量长文本做微调。Llama 3 就是从 8K 扩展到 128K 的。这比直接在长上下文上训练要高效得多，因为长序列的训练成本是 O(n²) 的。

序列并行解决了另一个工程问题：当序列长到单张 GPU 放不下时，需要把序列切分到多张 GPU 上。Arctic Ulysses 用 all-to-all 通信实现序列并行；ZeCO 则利用线性注意力的特性实现零通信的序列并行——这是线性注意力的又一个优势。

结合时的三重挑战

Three Tensions
挑战 01

KV 缓存膨胀

KV cache ∝ $n \times L \times 2n_h d_h$，与 MoE 无关但随序列线性爆炸。128K ctx ≈ 128 GB，超出单 GPU。

解法：GQA 降低头数 · MLA 压缩潜在空间 · 量化 fp8/int4
挑战 02

路由独立性

每 token 独立路由，长序列中语义连续的相邻 token 可能分发到截然不同的专家。语义连贯性完全依赖 Attention。

解法：共享专家为所有 token 提供通用基础处理
挑战 03

All-to-All 通信

分布式训练中专家分散在不同节点；每层均需 all-to-all 发送/接收 token，32 层连续 → 显著增加延迟。

解法：Expert Parallel + Pipeline Parallel · 通信与计算 overlap
MoE 扩大参数容量，长上下文扩大信息容量，但二者叠加的显存与通信压力倍增，需要专门的架构创新来抵消

【35:45 – 37:30 · 三重挑战】

三个挑战对应三张卡片。

挑战一：KV 缓存膨胀。KV cache 的大小正比于序列长度，与 MoE 无关——不管你用 Dense 还是 MoE，Attention 层的 KV cache 都一样大。128K 上下文下 KV cache 约 128GB，超出单张 GPU 的显存。解法包括 GQA 降低 KV 头数、MLA 压缩到低秩潜在空间、以及 fp8/int4 量化。

挑战二：路由独立性。MoE 中每个 token 独立路由，长序列中语义连续的相邻 token 可能被分发到截然不同的专家。比如一个句子的主语和谓语可能由不同专家处理，语义连贯性完全依赖 Attention 层来维持。共享专家在一定程度上缓解了这个问题——所有 token 都经过共享专家，获得通用语义基础。

挑战三：All-to-All 通信。分布式训练中专家分散在不同节点，每一层都需要 all-to-all 通信来发送和接收 token。一个 32 层的模型就需要 32 次 all-to-all，通信开销显著。解法是 Expert Parallel 和 Pipeline Parallel 的组合，以及通信与计算的 overlap。

底部的总结：MoE 扩大参数容量，长上下文扩大信息容量，但二者叠加的显存与通信压力倍增。DeepSeek-V2 的价值在于它同时处理了这三个问题。

【37:30 – 39:30 · DeepSeek-V2/V3 案例】

DeepSeek-V2 是目前 MoE + 长上下文结合的最佳案例。总参数 236B，激活 21B，上下文 128K。V3 进一步扩展到 671B 总参数、37B 激活，并采用 FP8 混合精度训练。

MLA（Multi-head Latent Attention）是 DeepSeek-V2 的另一个核心创新，专门解决 KV cache 膨胀问题。思路是：不直接缓存 K 和 V，而是把 hidden state 通过一个下投影矩阵压缩成低秩向量 c_KV（只有 512 维），只缓存这个压缩向量。解码时再通过上投影矩阵还原出 K 和 V。

右侧的流程图展示了这个过程：hidden state（d 维）→ 压缩到 c_KV（512 维）→ 解码时还原 K 和 V。缓存量从标准 MHA 的约 576 维降到约 38 维（考虑 GQA 等效），节省 93%。右下角的柱状图直观展示了这个压缩比。

工程上有一个关键技巧：推理时把上投影矩阵与后续的 Query 投影矩阵合并（矩阵吸收），消除了额外的上采样计算开销。这使得 MLA 的推理延迟不高于标准 GQA，但 KV cache 大幅减少。

训练成本方面，DeepSeek-V2 的训练成本约为同规模 Dense 模型的十分之一。V3 的 FP8 混合精度训练进一步降低了成本——据报道 V3 的完整训练成本约 550 万美元，远低于同级别模型。

左侧四条关键收获是今天报告的核心信息，我快速过一遍。第一，MoE 解耦了参数量和计算量，以 Dense 的推理成本撬动数倍的参数容量。第二，长上下文的核心瓶颈是三个：O(n²) 计算、KV cache 膨胀、Attention Distractions。第三，稀疏注意力（MoBA、NSA）将 MoE 的稀疏激活思想迁移到了注意力空间——这是两个领域的深层联系。第四，MoE 和稀疏注意力作用于 Transformer 的不同组件，加速效果是乘法叠加。

假说一：专家真的「专业化」了吗？有探针实验发现部分专家对特定语言或句法有统计偏好，但换一个随机种子重新训练，分配模式就完全打乱了。如果专业化只是统计巧合而非因果关系，那 MoE 的可解释性优势可能是假象。

假说二：长上下文真的被用到了吗？我们刚才看到的 Lost-in-the-Middle 效应说明，即使模型「支持」百万上下文，中间段的信息利用率仍然显著低于首尾。「长上下文」不等于「有效利用长上下文」。

假说三：MoE + 长上下文是协同还是叠加？两者的动机是独立的，目前缺乏严格的消融实验证明组合优于等参数量的 Dense 方案。系统复杂度倍增带来的工程成本是否值得？这是一个开放问题。
